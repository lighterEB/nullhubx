# NullHubX 前端视觉通用规则

日期：2026-03-30

参考架构：
- `docs/config-visualization/phase-06-backend-driven-ui-architecture.md`
- `docs/config-visualization/stage-a-shell-and-routing.md`

## 目标

这份文档定义 NullHubX 前端后续开发时默认遵循的视觉与排版规则，用来约束：

1. 页面壳层和信息架构不漂移
2. 字体、颜色、卡片、表单、表格等基础视觉不继续分叉
3. 后续 FE-05 / FE-06 / 新页面开发时有统一判断标准

本文件只定义“通用规则”，不替代具体页面交互设计，也不替代业务需求。

## 产品定位

NullHubX 的前端应被设计为：

1. 后端驱动的管理中枢
2. 多实例控制台
3. 编排与配置工作台

不应被设计为：

1. 营销官网式 SaaS 首页
2. 大面积炫技型展示页面
3. 过度终端风格的纯黑客主题 UI

一句话原则：`control plane` 优先，不做“泛模板后台”，也不做“概念海报站”。

## 默认视觉方向

在没有单独选定主题前，默认采用：

`Industrial Console x Soft Cyberpunk`

定义：

1. 结构秩序来自 `Industrial Console`
2. 现代感、背景纹理、状态辉光来自 `Soft Cyberpunk`
3. 赛博感来自层级、边框、发光、网格和动效，不来自满屏霓虹与大面积 `mono`

### 视觉基线

1. 主产品气质：未来感控制平面
2. 页面风格：深色中性底 + 工业面板层
3. 强调方式：低面积霓虹点缀，不做大面积高饱和铺色
4. 阅读优先级：`Settings / Config / Resources` 比 `Logs / Orchestration` 更克制

### 默认字体候选

1. 首选：`IBM Plex Sans SC` + `IBM Plex Sans` + `IBM Plex Mono`
2. 备选：`Noto Sans SC` + `Geist Sans` + `Geist Mono`

### 显示字体限制

如需更强赛博气质，允许引入显示字体，但只用于：

1. 品牌字
2. 少量英文 section heading
3. 特殊活动页或编排可视化标题

显示字体不得承担中文正文、中文标题、表单、表格或设置页主阅读任务。

未明确切换前，不再扩大 `DM Sans + JetBrains Mono` 的使用范围。

## 字体规则

### 0. 中文字体策略

中文 UI 不允许依赖英文字体 fallback 撑完整站。

默认策略：

1. 中文主字体：`IBM Plex Sans SC` 或 `Noto Sans SC`
2. 英文主字体：`IBM Plex Sans` 或 `Geist Sans`
3. 技术字体：`IBM Plex Mono` 或 `Geist Mono`

推荐默认栈：

1. `--font-sans: "IBM Plex Sans SC", "IBM Plex Sans", "Noto Sans SC", "PingFang SC", "Microsoft YaHei", sans-serif`
2. `--font-mono: "IBM Plex Mono", "JetBrains Mono", "SFMono-Regular", monospace`

规则：

1. 中文标题和正文必须由中文字体直接承担，不能落到系统随机 fallback
2. 英文展示字体最多做辅助，不承担中文主阅读场景
3. 如引入 `Space Grotesk` 一类显示字体，只能做英文品牌和局部英文标题

### 1. 主次分工

1. 页面标题、区块标题、正文、表单标签统一使用 `sans`
2. `mono` 只用于技术性信息，不用于主标题和大段说明文案

### 2. `mono` 的允许场景

1. 实例名：`component/name`
2. 端口、版本号、计数器
3. 状态标签、命令、路径、日志、JSON
4. 少量导航辅助标签或徽标

### 3. `mono` 的禁止场景

1. `h1/h2/h3` 主标题
2. 页面副标题
3. 空状态说明文案
4. 表单说明文字
5. 长段正文

### 4. 字号基线

1. 正文基线：`15px-16px`
2. 次级说明：`13px-14px`
3. Caption / 辅助标注：`12px`
4. 页面主标题：`28px-32px`
5. 区块标题：`18px-22px`

除状态栏这类极小信息位外，不再默认使用 `11px` 以下字号。

### 5. 中英混排

1. 中文环境下，优先保证汉字密度和行高稳定
2. 中英混排时，标题与正文仍保持同一 `sans` 家族，不做中文一套、英文一套的强割裂
3. `mono` 仅用于技术片段，不把整句中英混排内容切成终端风

## 颜色规则

### 1. 配色结构

每个主题必须只有以下 4 类颜色层级：

1. 中性色：背景、边框、正文
2. 单一主强调色：按钮、选中态、主链接、焦点态
3. 语义色：成功、警告、错误、信息
4. 少量辅助色：只允许用于个别模块，不进入全局基础 token

### 2. 限制

1. 全局主强调色只能有一个，不允许按钮、链接、选中态各用一套强调色
2. 不再把紫蓝系大面积铺满所有卡片、tab、边框和阴影
3. 语义色必须稳定：
   - 成功：绿
   - 警告：琥珀/黄
   - 错误：红
   - 信息：蓝或青

### 2.1 默认主题配色基线

默认主题按以下角色分配：

1. 主背景：深蓝黑或深石墨，不用纯黑
2. 面板背景：比主背景略亮一层
3. 主强调色：`cyan`
4. 次强调色：`violet`
5. 危险强调：`rose/red`

规则：

1. `cyan` 用于主按钮、激活态、焦点态、关键可操作元素
2. `violet` 只用于二级高亮、图谱节点、少量辅助标记
3. `magenta/rose` 不作为全站主强调色，只用于危险动作或少量赛博点缀
4. 不允许把 `cyan + violet + magenta` 同时大面积铺满一个页面

### 3. 背景策略

1. 页面背景以中性深色分层为主，不用纯黑
2. 可保留轻微纹理或网格，但透明度必须足够低，不能抢层级
3. 内容层靠边框、卡片底色、留白和局部发光区分，不靠大面积饱和底色区分
4. `Config / Settings / Resources` 需要比 `Logs / Orchestration` 更高对比、更低发光

## 页面壳层规则

### 1. 全局壳层

统一保持：

1. 顶部导航栏
2. 内容主区
3. 底部状态栏
4. 全局 toast 容器

新增页面不得绕过全局壳层重新定义独立导航体系。

### 2. 一级入口

一级入口按当前 IA 保持为：

1. `Dashboard / Overview`
2. `Instances`
3. `Resources`
4. `Orchestration`
5. `Settings`

命名必须全局一致。后续代码和文案只能选定一套，不允许继续混用：

1. `Dashboard` 或 `Overview`
2. `Resources` 或 `Connections`

### 3. 页面宽度

1. 常规内容页宽度控制在 `1160px-1400px`
2. 工作区页允许全宽，但必须有明确的区域分栏
3. 页面内卡片、列表、表单区块需要对齐同一内容列

### 4. Tab 放置规则

Tab 位置按信息层级决定，不按个人偏好决定。

默认规则：

1. 一级导航：顶部
2. 实例详情页二级 Tab：顶部
3. 筛选树、组件树、目录式导航：侧边
4. 复杂设置分组、复杂配置分组、Agents 内部子工作区：必要时使用侧边子导航

禁止项：

1. 全局已经有 `TopBar` 时，再把主工作区 Tab 全改成侧边，形成双重主导航
2. 在已有左侧筛选树的页面里，再叠加第二套强势左侧 Tab，导致结构过重
3. 为了“更像 IDE”而把所有二级页面强行改成侧边 Tab

NullHubX 当前推荐：

1. 一级入口保持顶部
2. `Instances` 页面保留左侧筛选/分组
3. 实例详情页固定顺序 Tab 保持顶部
4. `Config / Agents / Settings` 的深层子分组才允许侧边化

## 信息架构与布局规则

### 1. `Instances` 工作区

应遵循：

1. 左侧：组件分组或筛选树
2. 中部：实例网格或列表
3. 右侧：实例快速预览抽屉

在右侧抽屉未补齐前，不要再添加新的临时侧边块破坏中区结构。

### 2. 实例详情页

保持：

1. 顶部实例头
2. 状态摘要卡
3. 运行默认值区
4. 固定顺序二级 Tab

不得随意调整 Tab 顺序。顺序以架构文档为准：

1. `Overview`
2. `Agents`
3. `Config`
4. `Logs`
5. `Usage`
6. `History`
7. `Memory`
8. `Skills`
9. `Integration`

### 3. `Agents`

`Agents` 是实例内工作区，不是一级导航产品。
后续新增 agent 能力必须优先放在实例详情页 `Agents` Tab 内扩展。

### 4. 动态 UI 模块

动态 UI 模块是架构能力，不是自由样式入口。

规则：

1. 外层容器必须遵循主应用的间距、边框、背景和状态反馈规范
2. 模块加载失败态、未安装态、等待态必须使用主应用统一语气和视觉
3. 模块嵌入不得破坏主页面信息层级

## 组件级规则

### 1. 卡片

1. 卡片以深色面板底 + 细边框为主
2. 阴影只做辅助，不做主要层级手段
3. 发光只作为 hover / active / selected 的附加层，不作为静态常驻样式
4. 卡片圆角全站控制在一套 token 内，不再出现每页各自定义圆角

### 2. 按钮

只保留三类基础按钮：

1. 主按钮：主强调色
2. 次按钮：浅底边框
3. 危险按钮：明确危险色

禁止为不同页面继续发明新的“半主按钮”“亮底按钮”“彩色功能按钮”。

补充：

1. 赛博感主要来自 hover / focus / active 的细微发光
2. 主按钮发光面积必须可控，不能把整个页面都做成发光按钮墙

### 3. 状态徽标

状态徽标必须统一映射：

1. `running`
2. `starting`
3. `restarting`
4. `stopped`
5. `failed`
6. `unknown`

颜色、大小、字重、圆角风格必须由统一组件控制，不允许页面自己手搓。

### 4. 表单

1. 输入框、开关、下拉、textarea 的高度和边框风格统一
2. 表单提示文案一律使用 `sans`
3. 错误提示优先贴字段展示，避免只靠全局 toast
4. 配置页和设置页优先建立“表单区块”结构，不做散点式字段摆放

### 5. 表格与列表

1. 表头、行高、边框、空状态样式统一
2. 数值列、ID 列、版本列可用 `mono`
3. 行 hover 只做轻微强调，不使用重色块

### 6. 空状态、错误态、提示态

1. 空状态优先使用页面内卡片或虚线容器
2. 后台轮询失败优先页面内 banner 或局部错误提示
3. toast 只保留给用户主动动作的结果反馈
4. 赛博主题下也必须保持错误、空状态和等待态的可读性，不能只剩装饰感

## 文案与 i18n 规则

1. 同一页面不得出现中英混杂
2. 导航、标题、按钮、空状态、aria-label 使用同一套命名
3. 技术字段可以保留英文原词，但解释性文案必须本地化
4. 不允许在组件内继续新增硬编码 UI 文案

## 实现规则

### 1. Token 收口

所有全局视觉 token 统一放在：

1. `ui/src/app.css`

包括：

1. 字体变量
2. 颜色变量
3. 阴影
4. 圆角
5. 间距
6. 过渡

### 2. 禁止项

1. 页面级新增独立字体族
2. 在组件内直接写大量十六进制颜色
3. 为单个页面单独定义一套按钮或卡片系统
4. 用局部样式覆盖全局 token 来制造“特例页面”

### 3. 允许项

1. 在 `app.css` 先补 token，再逐页替换
2. 为编排、日志等特殊模块增加少量局部 token
3. 在不破坏全局基线的前提下，对工作区页面做更强的结构化布局

## 后续改版优先级

建议按下面顺序推进，而不是全站同时重做：

1. 全局字体与 token
2. `TopBar / StatusBar / 按钮 / 卡片 / 表单 / 表格`
3. `Dashboard`
4. `Instances`
5. `Instance Detail`
6. `Resources`
7. `Orchestration`
8. `Settings`

## 评审清单

每次前端样式改动都至少自查以下问题：

1. 这个页面看起来像“管理中枢”，还是像“营销页”？
2. 主标题是否还在滥用 `mono`？
3. 是否新增了新的强调色或新的按钮变体？
4. 是否出现局部页面自造 spacing / radius / shadow？
5. 导航、页面标题和路由命名是否一致？
6. 错误反馈是否做到了“主动动作 toast、被动状态内联”？
7. 新组件是否能复用到其它工作区，而不是一次性样式？

## 当前执行建议

在未正式定稿新主题前，先按以下最低要求收口：

1. 把标题系统从 `mono` 切回中文可用的 `sans`
2. 把 `mono` 缩回到实例名、状态、端口、日志、JSON
3. 把默认字体基线升级为 `IBM Plex Sans SC / IBM Plex Sans / IBM Plex Mono`
4. 把主背景与面板背景改成深色分层，而不是纯黑或浅色 SaaS 底
5. 收紧主强调色使用范围，以 `cyan` 为主、`violet` 为辅
6. 统一 `Dashboard/Overview` 与 `Resources/Connections` 命名
7. 在 FE-06 中把样式漂移问题和 `any` warning 一起收口
